# Проєкт 04: Зворотний зв'язок від runtime та структурний контроль

Впровадження спостережуваності runtime та перевірок структурних меж під час налагодження вбудованого дефекту runtime.

## Довідник по директоріях

| Директорія | Значення |
|------|------|
| `starter/` | **Відправна точка**: базується на рішенні P3, функції логування та структурних меж ще не реалізовано. `IndexingService` містить прихований вбудований баг: файли довжиною більше 1000 символів можуть генерувати порожні чанки. Немає скрипту перевірки архітектури та фінального чеклиста чистого стану. |
| `solution/` | **Еталонна реалізація**: модуль структурованого логування, скрипт перевірки архітектурних меж і виправлений вбудований баг. |

## Як використовувати

```sh
cd starter
npm install
# 1. Спостерігайте, чи може агент знайти баг через логи
# 2. Імпортуйте великий файл і перевірте, чи коректно відбувається чанкінг

cd ../solution
npm install
# Порівняйте, як структуровані логи прискорюють діагностику
```

## Точний контракт завдання

Проєкт 04 — це вправа з налагодження та встановлення захисних бар'єрів. Стартер вже містить
продуктовий зріз Проєкту 03, але runtime важче інспектувати, а вбудований
дефект чанкінгу слід виправляти лише після того, як агент побачить достатньо доказів.

| Функція / артефакт | Стан стартера | Докази рішення |
|------|------|------|
| Структуроване логування | Немає спільного сервісу логера | `src/services/logger.ts`, виклики логування у `main.ts`, `ipc-handlers.ts` та сервісах |
| Діагностика імпорту / індексування | Збої важко відстежити за виводом runtime | Записи логів навколо імпорту, початку/завершення індексування та шляхів збою QA |
| Архітектурні межі | Немає скрипту для перевірки дрейфу меж renderer/main/service | `scripts/check-architecture.sh`, `docs/ARCHITECTURE.md`, правила меж у AGENTS |
| Вбудований баг чанкінгу | Великі файли можуть генерувати некоректний/порожній вивід чанків | Виправлена логіка параграфів/чанків у `src/services/indexing-service.ts` |
| Чисте передавання | У стартері немає фінального чеклиста | `clean-state-checklist.md` |

Використовуйте довгий зразок документа для відтворення бага до та після виправлення. Успішне
рішення має демонструвати як зміни коду, так і діагностичні докази, а не лише
твердження про проходження.

## Охоплені функції

- Логи запуску
- Логи імпорту та індексування
- Видимий шлях збою QA
- Явні межі між шарами main, preload, renderer та services
- Налагодження вбудованого дефекту runtime
