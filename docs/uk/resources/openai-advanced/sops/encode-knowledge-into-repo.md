# SOP: Фіксація прихованих знань у репозиторії

Використовуйте цей SOP, коли важливий контекст досі живе в Google Docs,
тредах чатів, тікетах або головах людей.

## Мета

Зробити знання, невидимі для агента, доступними для виявлення в кодовій базі,
щоб нова сесія могла діяти на їх основі без опори на попередні розмови.

## Сигнали-тригери

- Агент продовжує запитувати, як працює система.
- Люди кажуть «ми вирішили це в Slack» або «дотримуйся того, що X сказав минулого тижня».
- Рев'ю посилаються на правила продукту або безпеки, яких немає у репозиторії.
- Нові сесії повторюють роботу виявлення, яка вже мала бути зафіксована.

## Виконання SOP

1. Перелічити невидимі джерела знань: документи, чати, негласні командні правила, усні рішення.
2. Для кожного джерела запитати: це архітектура, поведінка продукту, політика безпеки,
   очікування надійності, контекст плану чи довідковий матеріал?
3. Зафіксувати у відповідному артефакті репозиторію:
   - архітектура -> `ARCHITECTURE.md`
   - поведінка продукту -> `docs/product-specs/`
   - обґрунтування дизайну -> `docs/design-docs/`
   - стан виконання -> `docs/exec-plans/`
   - повторювані зовнішні посилання -> `docs/references/`
   - очікування якості або надійності -> `docs/QUALITY_SCORE.md` або `docs/RELIABILITY.md`
4. Замінювати розмиті формулювання операційно корисними.
5. Видаляти або позначати як застарілі дублікати, щоб репозиторій зберігав одну
   знайдену істину.

## Правила якісної фіксації

- Писати для виявлення, а не для літературної повноти.
- Надавати перевагу коротким документам із зрозумілими назвами файлів.
- Зв'язувати пов'язані артефакти між собою.
- Зберігати стійкі правила, а не протоколи нарад.
- Оновлювати репозиторій у тій самій сесії, в якій приймається рішення.

## Визначення завершення

- Свіжий агент може знайти відповідне правило без запиту до людини.
- Один і той самий факт не розкиданий по кількох суперечливих файлах.
- Новий артефакт розташований поруч із кодом або процесом, який він регулює.
